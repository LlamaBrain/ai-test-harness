# Captain SDLC — Cross-tool Trace Schema
Updated: 2026-04-08
Version: 0.1.4

Created: 2026-04-08

Planning doc for the **Pipeline Trace** (Seam 1 in `vision.md`) — the shared event log that links activities across interrogate, ATH, claude-release, and any future tools, so the pipeline can answer questions like "which design decision introduced this regression."

**Status: Planning.** Nothing here is implemented. Treat all claims as proposals open to argument.

## Goal

One question the schema must answer well, in priority order:

1. **Backward walk:** "Starting from this regression / bug / behavior change, walk the chain back to the design decision that caused it."
2. **Forward walk:** "Given this design decision, find every downstream artifact that was produced because of it" — commits, smoke runs, releases, post-ship issues.
3. **Pivoted views:** "Show me everything that happened against this commit / release / task across all tools."

If the schema answers (1) cleanly and (2)/(3) with modest tooling, it's earning its keep.

## Non-goals

- **Not a replacement for any tool's internal state.** Tools keep their own state; traces are the *interlink* between tools, not authoritative storage.
- **Not a metrics / monitoring system.** Traces are pipeline events, not telemetry samples. Regression envelope data (perf, alloc) belongs in ATH's own storage, *referenced* from a trace event.
- **Not a transport.** Traces are append-only files. They don't push notifications, trigger runs, or RPC between tools. Coordination stays in each tool's existing mechanisms.

## Format

**JSON Lines (`.jsonl`). One event per line. Append-only.**

Rationale:
- Every tool already knows how to emit JSON.
- Append-only avoids merge conflicts when multiple tools write concurrently.
- Line-orientation makes streaming reads cheap and partial files recoverable from a crash mid-write.
- Trivially diff-able and grep-able.

## Event envelope

Every event is one JSON object with this shape:

```json
{
  "schema_version": 1,
  "event_id": "uuid-v4",
  "timestamp": "2026-05-28T17:45:12.234Z",
  "tool": "ath",
  "tool_version": "0.1.0-preview.2",
  "kind": "ath.smoke.completed",
  "refs": {
    "project": "before-the-shade",
    "commit": "26e6d1a",
    "release": null,
    "design_doc": "docs/canonical/death-rewind.md",
    "task_id": "BTS-142"
  },
  "parents": ["6f1a..."],
  "links": [
    {"event_id": "a3c2...", "relation": "verifies"},
    {"event_id": "b8e0...", "relation": "regresses"}
  ],
  "payload": { /* kind-specific, see below */ }
}
```

### Required fields

| Field | Type | Purpose |
|---|---|---|
| `schema_version` | int | Bumped on breaking envelope change. Tools refuse unknown versions. |
| `event_id` | UUIDv4 | Globally unique. Generated by emitter. |
| `timestamp` | ISO 8601 UTC | When the event occurred. |
| `tool` | enum string | Registered tool identifier. |
| `tool_version` | semver | Emitting tool's version. Bumped on payload schema change. |
| `kind` | dotted string | Event kind (taxonomy below). |
| `refs` | object | Canonical references that situate this event. |

### Optional fields

| Field | Type | Purpose |
|---|---|---|
| `parents` | `event_id[]` | Direct causes. Smoke run's parent is the commit-completion event. |
| `links` | `{event_id, relation}[]` | Typed cross-references. Many-to-many, labelled. |
| `payload` | object | Kind-specific structured data. Schema per kind in the emitting tool's docs. |

## Event kinds (initial taxonomy)

Namespaced by phase. Additive — new kinds don't bump `schema_version`.

### Design (interrogate)
- `design.doc.created`
- `design.doc.updated`
- `design.task.scoped`
- `design.constitution.updated`
- `design.constitution.violation_detected` — Seam 6 constitution checker. Links to the constitution rule that was violated and the code location.
- `design.code_drift.detected`
- `design.gap.exposed`

### Code (claude-release at commit time)
- `code.commit.created` — emitted as part of commit finalization. Links to implemented tasks via `links[]` with relation `implements`.

### QA (ATH)
- `ath.smoke.started`
- `ath.smoke.completed` — payload includes pass/fail, envelope summary, link to artifacts.
- `ath.regression.detected` — links to the baseline run and the current run.
- `ath.replay.captured`
- `ath.replay.replayed`
- `ath.artifact_diff.computed`

### Release (claude-release)
- `release.gate.evaluated` — payload includes each gate's verdict and the inputs considered.
- `release.gate.summary` — aggregate verdict across all gates evaluated for a release.
- `release.gate.override` — when one or more gates are explicitly overridden. Records gate names and reasons.
- `release.published`
- `release.dependency_audit.completed`

### CICD (proposed)
- `cicd.build.started`
- `cicd.build.completed`
- `cicd.deployed`

### Live Ops (proposed)
- `liveops.crash.received`
- `liveops.feedback.received`
- `liveops.perf_sample.received`
- `liveops.telemetry.received`
- `liveops.task.created` — when an ingested item becomes a backlog task. Links back to the ingestion event via `triggers`.
- `liveops.task.occurrence` — when an additional ingestion event matches an existing task (dedup auto-link).
- `liveops.task.split` — when a merged task is split into separate tasks after re-triage.
- `liveops.task.merge` — when two independent tasks are discovered to be the same and merged.
- `liveops.task.recurrence` — when a closed task receives a new ingestion event matching it.
- `liveops.ingest.failed` — when an ingestor's source is unreachable or the ingestor errors.

### Marketing (proposed)
- `marketing.changelog.generated` — patch notes / player-facing changelog. Links to the artifact diffs it summarized.
- `marketing.assets.harvested`
- `marketing.published`

### Contract testing (Seam 4)
- `contract.test.failed` — when a Seam 4 contract test fails. Payload identifies the schema, the emitter, and the violation.

### Schema meta
- `trace.schema.unknown` — emitted by a consumer when it encounters an event whose `schema_version` it doesn't recognize. Self-referential entry for surfaceability.

Per-tool docs own each kind's payload schema. This doc owns only the envelope and the kind names.

## Linking conventions

The backward-walk goal implies the most common chain:

```
liveops.crash.received
  ↑ parent
ath.regression.detected
  ↑ parent
ath.smoke.completed (the failing run)
  ↑ parent
code.commit.created
  → links[implements] →
design.task.scoped (one or more)
  ↑ parent
design.doc.created
```

Conventions:

- **`parents` for direct causation.** Use sparingly — only when the parent event must have happened for this event to exist. A smoke run's parent is the commit it tested. A regression's parent is the smoke that detected it.
- **`links` for soft / many-to-many relations.** A commit `implements` tasks. A release `verifies` a set of smoke runs. A marketing changelog `summarizes` artifact diffs. Relations are named strings; the open relation set is documented per emitter.
- **No bidirectional links.** Events are append-only; if event A is created before event B, A can't link to B. B links back to A. (If A genuinely needs to know about B's existence, that's a sign A shouldn't have been emitted yet, or B is the wrong shape.)

### Established relations (initial)

| Relation | Meaning |
|---|---|
| `implements` | Commit implements task; release implements scoped batch. |
| `verifies` | Smoke run verifies a task; release verifies smoke runs. |
| `regresses` | Current run regresses a prior baseline. |
| `summarizes` | Marketing changelog summarizes artifact diffs. |
| `triggers` | Live ops event triggered a backlog task. |
| `replaces` | Updated design doc replaces a prior version. |
| `fixes` | Commit fixes a prior crash / regression event. |

Tools may introduce new relations. Consumers tolerate unknown relations.

## Identity and provenance

- `event_id` is globally unique. UUIDv4 is enough; no central coordinator.
- `tool` + `tool_version` identify the emitter. Tool authors bump `tool_version` when payload schemas change.
- No `machine_id` initially. Single-machine assumption. **Re-evaluate when CICD lands** — once a build runner emits events, traces become cross-machine and merging becomes a real concern.

## Storage and location

**Per-project trace directory:** `.captain-sdlc/trace/` at the project root (the repo where work is happening, not necessarily the Captain SDLC nerve-center repo).

**File naming:** `YYYY-MM-DD.jsonl`. New file daily. All tools append to the current day's file.

**Concurrent writes:** Append-only `O_APPEND` writes from multiple tools on POSIX are atomic per-line; Windows behavior is platform-specific. If contention surfaces, add per-tool sub-files (`YYYY-MM-DD.<tool>.jsonl`) and union at read time. Don't pre-optimize.

### Git policy — TBD

Two options:

| Option | Pro | Con |
|---|---|---|
| **`.gitignored`** | Local state, no git noise. Simplest. | Lost across machines; can't walk history from a fresh clone. |
| **Committed** | Source of truth; full history available from clone. | Adds noise to git; merge conflicts possible (mitigated by append-only + per-day files). |

**Lean: `.gitignored` initially**, with a separate archival mechanism (rsync to bucket, periodic squash-and-commit, whatever fits) added later if cross-machine traces become load-bearing. Reversible decision — switching from gitignored to committed is just `git add`.

### Retention

None initially. Manual rotation if files grow. Re-evaluate when accumulated traces actually hurt.

## Versioning policy

- **`schema_version`** governs the envelope fields described above.
- **`tool_version`** governs each event kind's payload.
- **Additive evolution within a `schema_version`.** New optional fields are fine. Renaming, removing, or changing types of existing required fields requires a `schema_version` bump.
- **Tools refuse unknown `schema_version`** and emit `trace.schema.unknown` rather than guessing. Surfaces drift visibly instead of silently.
- **Tools tolerate unknown event kinds and unknown relations.** Forward-compat without coordination.

## Privacy and sensitive payloads

If `liveops.crash.received` ever carries stack traces, user IDs, or save-game contents, traces become sensitive. **Before live ops ingestion lands**, we need:

- A redaction convention (e.g., a `redacted: true` marker on fields the emitter scrubbed).
- A policy on whether sensitive payloads live in the trace at all, vs sit in a side-store referenced by URI.
- A retention policy for sensitive data distinct from the general trace.

Defer the policy until live ops ingestion is actually scoped — but flag it now so the design doesn't paint into a corner.

## Idempotency

If a tool re-runs and re-emits the same logical event (e.g., the smoke harness reruns the same smoke against the same commit), the schema says: **re-emit.** Each emission gets a new `event_id`. Append-only means dedup-at-write is impossible anyway. Consumers that care about deduplication do so at read time using `(tool, kind, refs)` as the dedup key, keeping the latest.

## Open scoping questions

1. **Schema evolution governance.** Who owns the canonical schema? Currently this repo (Captain SDLC nerve center). When/if the cross-tool layer becomes its own tool, ownership moves.
2. **Cross-machine merging.** Deferred. The single-machine assumption breaks when CICD lands. Worth planning the merge model (timestamp-merge? per-machine subdirs?) before that ships.
3. **A minimal consumer milestone.** Schema-first is the right discipline, but a *minimal consumer* (e.g., a CLI that walks the chain backward from an `ath.regression.detected` event and prints the implicated design doc) would prove the schema actually answers the priority-1 goal. Worth building as a milestone alongside the first emitter, not after.
4. **Privacy model for live ops payloads.** See above.
5. **Trace as a single file vs per-tool subfiles.** Lean: single file per day, swap to per-tool if concurrent-write contention shows up. Don't pre-optimize.
6. **`refs.project` namespacing.** If traces from multiple projects ever co-mingle (unlikely but possible — Captain SDLC tooling itself spans repos), `project` needs to be unambiguous. Probably a short slug agreed in the project's `package.json` / equivalent.

## Minimal first cut

If we ship the smallest useful version:

- **Envelope only.** No initial event-kind taxonomy enforcement — tools emit whatever `kind` they want; consumers tolerate unknowns.
- **One emitter.** ATH, since it's the nerve center and already produces the events most likely to need explaining ("why did this smoke fail?").
- **One consumer.** A CLI: `captain-trace walk <event_id>` prints the parent chain. `captain-trace by-commit <sha>` lists all events against a commit.

That proves the format, surfaces the gaps, and lets other tools come online incrementally. Taxonomy formalization and additional emitters follow.

## What this doc owns vs what tool docs own

- **This doc owns:** envelope, format, storage location, versioning policy, linking conventions, the event-kind name registry.
- **Tool docs own:** payload schemas for each kind their tool emits, the specific relations they use in `links[]`, the side-store URIs they reference.

Per-tool docs should link back to this one as the canonical envelope reference.
